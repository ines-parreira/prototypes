# implement-stats-scope

Implement a new stats scope in the frontend from a Linear ticket. The ticket must contain a scope definition block (generated by `/define-scope-for-reporting-query` in the backend repo).

## Step 1: Parse Input

Extract the Linear ticket ID from the argument:
- URL format: `https://linear.app/gorgias/issue/ANALYT-5728/...` → ticket ID is `ANALYT-5728`
- If no URL provided, ask the user for a Linear ticket URL

## Step 2: Fetch the Ticket

Call `mcp__claude_ai_Linear__get_issue` with the extracted ticket ID.

Find the scope definition block in the ticket description. It looks like:
```
{
    scope: MetricScope.SomeName,
    measures: ['measureName'],
    dimensions: ['dimName', 'channel', 'ticketId'],
    timeDimensions: ['eventDatetime'],
    filters: ['filterA', 'filterB', 'periodEnd', 'periodStart'],
    order: ['measureName', 'eventDatetime'],
}
```

If no scope definition is found in the ticket, stop and tell the user:
> No scope definition found. Please run `/define-scope-for-reporting-query` in the Gorgias backend repository first, then link the resulting config block to this ticket.

Parse out:
- `scopeEnumName`: the PascalCase name after `MetricScope.` (e.g., `AiAgentCoverageRate`)
- `scopeSlug`: derived from the enum value string (e.g., `'ai-agent-coverage-rate'`)
- `scopeName`: camelCase version of the scope (e.g., `aiAgentCoverageRate`)
- `measures`, `dimensions`, `timeDimensions`, `filters`, `order` arrays

## Step 3: Read Existing State

Read these files **in parallel**:

| File | Purpose |
|------|---------|
| `apps/helpdesk/src/domains/reporting/hooks/metricNames.ts` | MetricScope enum + METRIC_NAMES_BY_SCOPE |
| `apps/helpdesk/src/domains/reporting/models/scopes/types.ts` | MeasureName, DimensionName, FilterName unions |
| `apps/helpdesk/src/domains/reporting/models/scopes/utils.ts` | createScopeFilters switch statement |
| `apps/helpdesk/src/core/flags/utils/newApiMetricFlags.ts` | Scope groups (P1_SCOPES, P2_SCOPES, P3_SCOPES, P5_AI_AGENT_REVAMP_SCOPES, …) tied to feature flags |

## Step 4: Identify Missing Items

Check for each item:

### MetricScope
Does `MetricScope.{scopeEnumName}` already exist in the enum? Note the enum location in the file (around line 579 in metricNames.ts).

Also check `METRIC_NAMES_BY_SCOPE`: since it is typed as `Record<MetricScope, MetricName[]>`, adding a new enum value **requires** a corresponding entry or TypeScript will error.

### Types
- **MeasureName**: are all items from `measures` already in the `MeasureName` union?
- **DimensionName**: are all items from `dimensions` already in `DimensionName`?
- **FilterName subtypes**: for each item in `filters` (excluding `periodStart`/`periodEnd` which are already handled), is it in `StringFilterName`, `NumberFilterName`, or `BooleanFilterName`?

For new filter names not yet in any subtype, guess the appropriate type using these rules:
- IDs: names ending in `Id` (e.g., `agentId`, `teamId`, `storeIntegrationId`) → `NumberFilterName`
- Boolean flags: names starting with `is` or `has` (e.g., `isAnsweredByAgent`) → `BooleanFilterName`
- Everything else → `StringFilterName`

**Present the full table to the user and ask if any guessed types are wrong before proceeding.** Example:

```
The following new filter names were found with guessed types:

| Filter Name        | Guessed Type     |
|--------------------|------------------|
| engagementType     | StringFilterName |
| storeIntegrationId | NumberFilterName |

Let me know if any of these are wrong, otherwise I'll proceed with these types.
```

### Scope group

Look at the existing groups in `newApiMetricFlags.ts` (`P1_SCOPES`, `P2_SCOPES`, `P3_SCOPES`, `P5_AI_AGENT_REVAMP_SCOPES`, etc.) and their associated feature flags.

Based on the ticket context (feature area, flag name mentioned in the ticket if any), make an educated guess about which group the new scope belongs to, then **ask the user to confirm** before proceeding:

```
Which scope group should MetricScope.{ScopeEnumName} be added to?

Available groups (from newApiMetricFlags.ts):
- P1_SCOPES → ReportingP1MetricMigration
- P2_SCOPES → ReportingP2MetricMigration
- P3_SCOPES → ReportingP3MetricMigration
- P5_AI_AGENT_REVAMP_SCOPES → ReportingP5AiAgentRevampQueryScopes

My guess: P5_AI_AGENT_REVAMP_SCOPES (based on [reason from ticket context])

Let me know if this is correct or which group to use instead.
```

### createScopeFilters switch cases
For each filter in the scope's `filters` array (excluding `periodStart`/`periodEnd`), does a `case '{filterName}':` already exist in the switch statement in `utils.ts`?

Note: some filters already exist in the catch-all block (lines ~267–309) that handles them with `statFilters[filterKey]` — if a filter is there, it does **not** need a new case.

## Step 5: Apply Changes (in order)

### 5a. Update `types.ts`

Add any missing type entries:
- Missing `MeasureName` values: add to the `MeasureName` union
- Missing `DimensionName` values: add to the `DimensionName` union
- Missing `StringFilterName` values: add to the `StringFilterName` union
- Missing `NumberFilterName` values: add to the `NumberFilterName` union
- Missing `BooleanFilterName` values: add to the `BooleanFilterName` union

### 5b. Update `metricNames.ts`

Add the new `MetricScope` enum entry in the appropriate section (e.g., P5 or wherever the related scopes are):
```typescript
{ScopeEnumName} = '{scope-slug}',
```

Also add a corresponding entry in `METRIC_NAMES_BY_SCOPE`:
```typescript
[MetricScope.{ScopeEnumName}]: [],
```

The array starts empty because the scope file won't define any query factories (those come later when charts are built on top of this scope).

### 5c. Update `utils.ts`

For each filter in the scope's `filters` array that lacks a switch case:

- **Simple string/number/boolean filters** (where `statFilters[filterKey]` works directly): add to the catch-all block. Example of the catch-all block:
  ```typescript
  case 'aiAgentSkill':
      {
          const filter = statFilters[filterKey]
          if (filter && hasFilter(filter)) {
              filters.push(
                  createStandardFilter(
                      filterKey,
                      filter.operator,
                      filter.values,
                  ),
              )
          }
      }
      break
  ```
  Add new cases before the closing `break` of the last existing case in the catch-all block.

- **Complex filters** (e.g., where the stat filter key differs from the filter member name, or requires transformation): add a dedicated `case` with the appropriate logic, following the patterns of similar cases like `agentId`, `teamId`, `channel`, etc.

### 5d. Update `newApiMetricFlags.ts`

Add `MetricScope.{ScopeEnumName}` to the scope group array confirmed by the user:

```typescript
export const P5_AI_AGENT_REVAMP_SCOPES: MetricScope[] = [
    MetricScope.OverallAutomationRate,
    MetricScope.{ScopeEnumName}, // ← add here
]
```

### 5f. Create the scope file

Create `apps/helpdesk/src/domains/reporting/models/scopes/{scopeName}.ts`:

```typescript
import { MetricScope } from 'domains/reporting/hooks/metricNames'
import type { Context } from 'domains/reporting/models/scopes/scope'
import { defineScope } from 'domains/reporting/models/scopes/scope'

export const {scopeName}Scope = defineScope({
    scope: MetricScope.{ScopeEnumName},
    measures: [{measures as quoted strings}],
    dimensions: [{dimensions as quoted strings}],
    timeDimensions: [{timeDimensions as quoted strings}],
    filters: [{filters as quoted strings}],
    order: [{order as quoted strings}],
})

export type {ScopeEnumName}Context = Context<typeof {scopeName}Scope.config>
```

Notes:
- Use the exact array values from the scope definition (keep all fields even if empty)
- Omit `dimensions`, `timeDimensions`, or `order` only if they were absent from the source definition
- Do NOT add query factories — those belong in separate feature files

### 5g. Create the test file

Create `apps/helpdesk/src/domains/reporting/models/scopes/tests/{scopeName}.spec.ts`:

The tests verify `createScopeFilters` behavior:
- Period filters are always present
- Each optional filter in the scope is included when provided in context, omitted when not provided

```typescript
import { createScopeFilters } from 'domains/reporting/models/scopes/utils'
import { {scopeName}Scope } from 'domains/reporting/models/scopes/{scopeName}'
import type { StatsFilters } from 'domains/reporting/models/stat/types'

describe('{scopeName}Scope', () => {
    const baseFilters: StatsFilters = {
        period: {
            start_datetime: '2025-09-03T00:00:00.000',
            end_datetime: '2025-09-03T23:59:59.000',
        },
    }

    it('includes period filters', () => {
        const result = createScopeFilters(baseFilters, {scopeName}Scope.config)

        expect(result).toContainEqual(
            expect.objectContaining({ member: 'periodStart', operator: 'afterDate' }),
        )
        expect(result).toContainEqual(
            expect.objectContaining({ member: 'periodEnd', operator: 'beforeDate' }),
        )
    })

    // For each optional filter in the scope (excluding periodStart/periodEnd):
    it('includes {filterName} filter when provided', () => {
        const filters: StatsFilters = {
            ...baseFilters,
            // Add the appropriate statFilters key for this filter
            // e.g., for 'channel': channels: { operator: 'one-of', values: ['email'] }
            // e.g., for 'aiAgentSkill': aiAgentSkill: { operator: 'one-of', values: ['support'] }
        }
        const result = createScopeFilters(filters, {scopeName}Scope.config)

        expect(result).toContainEqual(
            expect.objectContaining({ member: '{filterName}', operator: 'one-of' }),
        )
    })

    it('omits {filterName} filter when not provided', () => {
        const result = createScopeFilters(baseFilters, {scopeName}Scope.config)

        expect(result).not.toContainEqual(
            expect.objectContaining({ member: '{filterName}' }),
        )
    })
})
```

To know the correct `statFilters` key for each filter, look at how the filter is handled in `utils.ts`:
- `case 'channel'` reads from `statFilters.channels`
- `case 'aiAgentSkill'` reads from `statFilters.aiAgentSkill`
- `case 'storeIntegrationId'` reads from `statFilters.storeIntegrations`
- etc.

Use `LogicalOperatorEnum.ONE_OF` (from `'domains/reporting/pages/common/components/Filter/constants'`) as the operator when constructing test filters that expect `'one-of'` in output.

## Key Conventions

- `scopeName`: camelCase (e.g., `aiAgentCoverageRate`) — used for variable names and filenames
- `ScopeEnumName`: PascalCase (e.g., `AiAgentCoverageRate`) — used for the enum entry and Context type
- `scope-slug`: kebab-case (e.g., `'ai-agent-coverage-rate'`) — used as the enum string value
- The exported scope variable is `{scopeName}Scope` (e.g., `aiAgentCoverageRateScope`)
- The exported context type is `{ScopeEnumName}Context` (e.g., `AiAgentCoverageRateContext`)
- Do not add comments to generated code unless there is complex non-obvious logic
